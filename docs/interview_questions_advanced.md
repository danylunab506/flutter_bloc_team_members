# Interview Questions — Advanced Topics

---

## Performance & DevTools

**1. How do you detect a performance problem in a Flutter app?**

> The first step is to open Flutter DevTools and use the Performance panel. I enable "Track widget rebuilds" to see which widgets are being rebuilt on each frame. If a frame takes longer than 16ms (60fps) or 8ms (120fps), it appears in red in the flame chart. There I identify whether the bottleneck is in the UI thread or the raster thread. If it's in the UI thread, I look for unnecessary rebuilds or heavy work in the widget tree. If it's in the raster thread, I look for expensive effects like `saveLayer`, complex shadows, or uncached images.

---

**2. What is the Widget Rebuild Tracker and when do you use it?**

> It's a tool inside the DevTools Inspector that highlights with colors the widgets that were rebuilt in the last frame. Green means few rebuilds, red means many. I use it when I suspect a widget is rebuilding more than necessary. For example, if a `Text` showing a counter is rebuilding together with the entire screen, I wrap it in a `BlocSelector` or split the widget into a smaller one to isolate the rebuild.

---

**3. What's the difference between the UI thread and the Raster thread in the flame chart?**

> The UI thread runs Dart code: widget tree construction, BLoC logic, animations. The Raster thread takes the layer tree generated by the UI thread and turns it into pixels using Skia/Impeller. If the UI thread is saturated, there's too much work in Dart. If the Raster thread is saturated, there are expensive graphical operations — for example, `BackdropFilter`, complex `ClipPath`, or very large uncompressed images. The solution for each is different.

---

**4. What is `const` and why does it matter for performance?**

> A `const` widget is built only once and reused. Flutter can skip its rebuild entirely because it knows its output never changes. If a parent widget rebuilds, `const` children are not touched. In this project all widgets that don't depend on state are `const` — `LoadingWidget`, `EmptyMembersWidget`, the `SizedBox`es, the static `Text`s. It's the cheapest optimization available: it only requires adding the keyword.

---

**5. How do you avoid unnecessary rebuilds in a long list?**

> Three main strategies:
> 1. Use `ListView.builder` or `SliverList` with `SliverChildBuilderDelegate` — it only builds visible items, not the whole list.
> 2. Extract each item into its own widget with `const` or with stable `keys` so Flutter can reconcile the tree efficiently.
> 3. Use `BlocSelector` or `buildWhen` so the list widget doesn't rebuild when parts of the state that don't affect it change.
> In this project I use `SliverList` with `SliverChildBuilderDelegate` precisely for this reason.

---

**6. What is `RepaintBoundary` and when do you use it?**

> `RepaintBoundary` creates a separate composition layer for its subtree. When that subtree repaints, Flutter doesn't have to repaint the rest of the screen. It's useful for widgets that animate or change frequently on an otherwise static screen — for example, a counter that updates every second, or a Lottie animation inside a list. The cost is extra memory for the additional layer, so it shouldn't be applied indiscriminately.

---

**7. How do you profile memory in Flutter DevTools?**

> In the Memory panel in DevTools I can take heap snapshots and compare two points in time. I look for objects that grow without bound — lists that accumulate, streams that aren't closed, BLoCs that aren't disposed. I also use "Allocation Tracing" to see which classes are being instantiated the most. A common memory leak pattern in Flutter BLoC is forgetting to call `bloc.close()` when unmounting the widget, but `BlocProvider` does this automatically.

## Clean Architecture

**8. What is the core principle of Clean Architecture?**

> The dependency rule: inner layers don't know about outer layers. The domain doesn't know that Flutter exists, or `get_it`, or `http`. The presentation doesn't know how the repository is implemented. Each layer depends only inward. This allows testing the domain with pure Dart, changing the datasource without touching the UI, and changing the state framework without touching the business logic.

---

**9. What goes in the domain and what doesn't?**

> What goes in the domain: entities (pure business objects), repository contracts (abstract interfaces), and use cases (business logic). Nothing that depends on Flutter, external packages, JSON, HTTP, or the UI goes in. If you need to import anything other than `dart:core` or your own entities, you're probably putting something in the wrong place.

---

**10. What are use cases for if they only delegate to the repository?**

> In simple projects they seem unnecessary, but they serve two purposes. First, they are the extension point: when logic grows — validations, combining multiple repositories, business rules — the use case absorbs that complexity without contaminating the BLoC or the repository. Second, they are the contract unit with the presentation: the BLoC doesn't know the repository directly, it only knows use cases with names that express intent (`GetTeamMembers`, `RemoveTeamMember`). That makes the BLoC more readable and easier to test in isolation.

---

**11. What's the difference between an entity and a model (scheme)?**

> The entity is the pure business object — it defines what a `TeamMember` is from the domain's perspective. It knows nothing about JSON or APIs. The scheme (in other projects it's called a "model" or "DTO") is the data representation of a specific source — it knows how to do `fromJson`, it can have extra fields the API returns but the domain doesn't need, or different field names. It lives in the `data` layer (in this project, in `data/schemes/`) and is converted into an entity before leaving the repository. In this project the `TeamMemberScheme` inherits from `TeamMember` because the source is local and there is no divergence; if a remote API with a different shape appeared, I would introduce a mapper in the repository and the scheme would stop inheriting from the entity.

---

**12. How would you test the repository in isolation?**

> By mocking the datasource. The repository receives the datasource via constructor (dependency injection), so in the test I pass a mock that returns controlled data. I verify that the repository correctly transforms what the datasource returns, applies the cache, and handles errors. Without mocks, the test would depend on reading a real file, which makes it slow and fragile.

---

## Dependency Injection

**13. What is dependency injection and why use it?**

> It's the pattern of passing a class's dependencies from outside instead of creating them inside. Instead of `TeamMembersBloc` doing `final repo = TeamMemberRepositoryImpl()`, it receives the repository via its constructor. This has three advantages: classes are testable (you can pass a mock), they are interchangeable (you can pass another implementation without changing the class), and the dependency graph is explicit and centralized.

---

**14. What's the difference between `registerSingleton`, `registerLazySingleton` and `registerFactory` in get_it?**

> `registerSingleton` creates the instance immediately when registered. `registerLazySingleton` creates it the first time it's requested and reuses it thereafter. `registerFactory` creates a new instance every time it's requested. In this project: datasource, repository and use cases are `lazySingleton` because they must be unique and don't need to exist from startup. The BLoC is `factory` because each screen needs a new instance with a clean state.

---

**15. What's the difference between Service Locator and constructor DI?**

> Service Locator (like `get_it`) is a global registry from which any class can request its dependencies by calling `getIt<MyClass>()`. Constructor DI is when dependencies are explicitly passed when creating the object. Service Locator is more convenient but hides dependencies — from the constructor signature you don't know what the class needs. Constructor DI is more explicit and more testable. In this project I use get_it as an orchestrator in the composition layer (router, DI modules), but the domain classes and the BLoC receive their dependencies via constructor — the best of both worlds.

---

**16. How would you handle scoped dependencies in get_it? For example, dependencies that only live while a screen is open.**

> get_it supports scopes with `pushNewScope()` and `popScope()`. When navigating to a screen, you open a new scope and register that screen's dependencies. On exit, you do `popScope()` and get_it discards them automatically. It's useful for expensive dependencies that don't need to live through the whole session. In this project it's not necessary because the BLoC is handled with `registerFactory` and `BlocProvider` takes care of the lifecycle, but in larger apps scopes are the scalable solution.

---

## Mixins

**17. What is a mixin in Dart and what is it for?**

> A mixin is a mechanism to reuse code between classes without inheritance. It's defined with `mixin` and applied with `on` or `with`. Unlike inheritance, a class can use multiple mixins. It's useful for sharing cross-cutting behavior — logging, validation, analytics — without creating artificial inheritance hierarchies.

---

**18. What's the difference between `mixin`, `abstract class` and `interface` in Dart?**

> An `abstract class` can have implementation and state, but a class can only extend one. A `mixin` cannot be instantiated directly and is applied with `with` — allowing multiple composition. In Dart there's no explicit `interface` keyword: any class or abstract class can be used as an interface by implementing it with `implements`, which forces you to implement all its members without inheriting implementation. In Clean Architecture, repository contracts are `abstract class`es used as interfaces with `implements`.

---

**19. When would you use a mixin in a Flutter project?**

> Common cases:
> - `WidgetsBindingObserver` as a mixin to detect app lifecycle changes (foreground/background).
> - A logging mixin that intercepts methods and logs calls.
> - A form validation mixin reusable across multiple BLoCs or cubits.
> - `AutomaticKeepAliveClientMixin` to keep a tab's state alive inside a `TabBarView`.

---

**20. What restriction does `on` impose on a mixin?**

> `on` restricts which classes the mixin can be applied to. For example:
> ```dart
> mixin LoadingMixin on Bloc {
>   void showLoading() => emit(LoadingState());
> }
> ```
> Only classes that extend `Bloc` can use `LoadingMixin`. This allows the mixin to access the base class's methods with type safety.

## Push Notifications — FCM / PubNub

**21. How would you implement push notifications with FCM in Flutter?**

> With the `firebase_messaging` package. The flow is:
> 1. Initialize Firebase with `Firebase.initializeApp()`.
> 2. Request permissions with `FirebaseMessaging.instance.requestPermission()`.
> 3. Get the device token with `getToken()` and send it to the backend to associate it with the user.
> 4. Configure handlers: `onMessage` for foreground notifications, `onMessageOpenedApp` for when the user taps the notification from background, and `getInitialMessage` for when the app was terminated.
> 5. To show notifications in foreground on iOS and Android, `flutter_local_notifications` is used because FCM doesn't show them automatically in foreground.

---

**22. What's the difference between foreground, background and terminated notifications?**

> — **Foreground**: the app is open. FCM delivers the message to `onMessage`. The notification is not automatically shown in the system — you have to show it manually with `flutter_local_notifications` if you want the user to see it.
> — **Background**: the app is in the background. FCM shows the notification automatically in the system. When tapped, `onMessageOpenedApp` is triggered.
> — **Terminated**: the app is closed. FCM shows the notification. When tapped, the app opens and `getInitialMessage()` returns the message that started it. It must be called in `main` after `Firebase.initializeApp()`.

---

**23. What specific configuration does FCM require on iOS that it doesn't require on Android?**

> On iOS you have to:
> 1. Enable the "Push Notifications" and "Background Modes > Remote notifications" capabilities in Xcode.
> 2. Upload the APNs Authentication Key or the APNs Certificate to the Firebase console.
> 3. Explicitly request permission from the user with `requestPermission()` — on iOS it's mandatory; on Android from Android 13 onward as well, but historically it wasn't.
> 4. In some cases configure `setForegroundNotificationPresentationOptions` to control whether the notification is shown as a banner, badge, or sound while the app is in foreground.

---

**24. How would you use PubNub instead of FCM? What's the main difference?**

> PubNub is a real-time messaging service based on publish/subscribe. The main difference with FCM is that PubNub is bidirectional and low-latency — it's designed for chat, presence, and real-time synchronization. FCM is optimized for operating-system push notifications, which the OS can batch or delay to save battery. In Flutter you use the Dart `pubnub` SDK. You subscribe to channels and receive messages in real time through a stream. For system push notifications with PubNub, you can configure the Mobile Push Gateway, which internally uses FCM/APNs.

## Sockets / WebSockets

**25. How would you implement WebSockets in Flutter?**

> Dart has native support via `WebSocket` in `dart:io`. In Flutter you can use it directly or the `web_socket_channel` package, which provides a more convenient stream-based API. The flow is:
> ```dart
> final channel = WebSocketChannel.connect(Uri.parse('wss://...'));
> channel.stream.listen((message) { /* receive */ });
> channel.sink.add('message');   // send
> channel.sink.close();          // close
> ```
> The stream integrates naturally with BLoC: the BLoC listens to the socket stream and emits states based on the received messages.

---

**26. How would you integrate WebSockets with BLoC?**

> The BLoC subscribes to the socket stream in its constructor or in response to an event, and emits states with the received data. It's important to cancel the subscription in the BLoC's `close()` to avoid memory leaks:
> ```dart
> class ChatBloc extends Bloc<ChatEvent, ChatState> {
>   final WebSocketChannel _channel;
>   late final StreamSubscription _sub;
>
>   ChatBloc(this._channel) : super(ChatInitial()) {
>     _sub = _channel.stream.listen(
>       (message) => add(MessageReceived(message)),
>     );
>   }
>
>   @override
>   Future<void> close() {
>     _sub.cancel();
>     _channel.sink.close();
>     return super.close();
>   }
> }
> ```

---

**27. How would you handle automatic reconnection if the socket drops?**

> By implementing a retry mechanism with exponential backoff. When the socket stream emits an error or closes unexpectedly, I wait an increasing amount of time before reconnecting (1s, 2s, 4s, up to a maximum). The BLoC emits a `Reconnecting` state so the UI can show it. I also listen to `ConnectivityStream` to reconnect automatically when the device recovers the network. It's important to have a retry limit and a `ConnectionFailed` state when attempts are exhausted.

---

**28. What's the difference between WebSockets and Server-Sent Events (SSE)?**

> WebSocket is bidirectional — client and server can send each other messages at any time over the same connection. SSE is unidirectional — only the server sends events to the client over a persistent HTTP connection. SSE is simpler to implement, works over HTTP/2, and the browser reconnects it automatically. WebSocket is necessary when the client also needs to send frequent messages. For read-only cases like news feeds or status updates, SSE is sufficient and lighter.

## Freezed

**29. What is Freezed and what is it for?**

> Freezed is a code generator for Dart that creates immutable classes with value equality, `copyWith`, pattern matching, and union types. It replaces the boilerplate of writing `==`, `hashCode`, `copyWith` and constructors by hand. It's especially useful for states, entities and models where immutability and value equality are important.

---

**30. In this project you used Equatable instead of Freezed. What's the difference?**

> Equatable requires you to manually declare the fields in `props` and gives you value equality and `toString`. Freezed generates everything automatically and also adds `copyWith` and enforced immutability. Equatable is lighter — it requires no code generation or `build_runner`. Freezed is more powerful but adds complexity to the setup. For this project Equatable is enough. In a project with complex models that need frequent `copyWith`, I would choose Freezed.

---

**31. How would you model the BLoC states with Freezed?**

> ```dart
> @freezed
> sealed class TeamMembersState with _$TeamMembersState {
>   const factory TeamMembersState.initial() = TeamMembersInitial;
>   const factory TeamMembersState.loading() = TeamMembersLoading;
>   const factory TeamMembersState.loaded(List<TeamMember> members) = TeamMembersLoaded;
>   const factory TeamMembersState.empty() = TeamMembersEmpty;
>   const factory TeamMembersState.error(String message) = TeamMembersError;
> }
> ```
> Freezed generates `==`, `hashCode`, `copyWith` and lets you use `when` / `maybeWhen` for pattern matching. The result is equivalent to this project's `sealed class` + Equatable, but with less manual code.

---

**32. What is `copyWith` and why is it important in immutable objects?**

> `copyWith` creates a new instance of the object with some fields modified, keeping the rest the same. It's the alternative to mutating an object directly. Instead of `member.name = 'new'` (mutation, forbidden on immutable objects), you do `member.copyWith(name: 'new')` which returns a new object. This is fundamental in BLoC — states are immutable and each change produces a new state, which makes the flow predictable and auditable.

---

**33. Freezed can be used for union types. Explain with an example.**

> A union type is a type that can be one of several variants, each with its own fields. That's exactly what BLoC states are. With Freezed:
> ```dart
> @freezed
> class ApiResult<T> with _$ApiResult<T> {
>   const factory ApiResult.success(T data) = Success;
>   const factory ApiResult.failure(String error) = Failure;
>   const factory ApiResult.loading() = Loading;
> }
> ```
> Then in the UI:
> ```dart
> result.when(
>   success: (data) => Text(data.toString()),
>   failure: (error) => Text(error),
>   loading: () => CircularProgressIndicator(),
> );
> ```
> The compiler guarantees that you handle all variants. It's the equivalent of this project's `sealed class` + `switch`, but generated automatically.
